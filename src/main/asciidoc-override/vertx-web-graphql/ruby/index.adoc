= Vert.x Web GraphQL

Vert.x Web GraphQL extends Vert.x Web with the https://www.graphql-java.com[GraphQL-Java] library so that you can build a GraphQL server.

TIP: This is the reference documentation for Vert.x Web GraphQL.
It is highly recommended to get familiar with the GraphQL-Java API first.
You may start by reading the https://www.graphql-java.com/documentation/v12/[GraphQL-Java documentation].

WARNING: This module has _Tech Preview_ status, this means the API can change between versions.

== Getting started

To use this module, add the following to the _dependencies_ section of your Maven POM file:

[source,xml,subs="+attributes"]
----
<dependency>
 <groupId>io.vertx</groupId>
 <artifactId>vertx-web-graphql</artifactId>
 <version>3.7.1</version>
</dependency>
----

Or, if you use Gradle:

[source,groovy,subs="+attributes"]
----
compile 'io.vertx:vertx-web-graphql:3.7.1'
----

== Handler setup

Create a Vert.x Web `link:../../yardoc/VertxWeb/Route.html[Route]` and a `link:../../yardoc/VertxWebGraphql/GraphQLHandler.html[GraphQLHandler]` for it:

[source,ruby]
----
require 'vertx-web-graphql/graph_ql_handler'
graphQL = setup_graph_ql_java()

router.route("/graphql").handler(&VertxWebGraphql::GraphQLHandler.create(graphQL).method(:handle))

----

The handler serves both `GET` and `POST` requests.
However you can restrict the service to one type of HTTP method:

[source,ruby]
----
require 'vertx-web-graphql/graph_ql_handler'
graphQL = setup_graph_ql_java()

router.post("/graphql").handler(&VertxWebGraphql::GraphQLHandler.create(graphQL).method(:handle))

----

TIP: The `link:../../yardoc/VertxWebGraphql/GraphQLHandler.html[GraphQLHandler]` does not require a `link:../../yardoc/VertxWeb/BodyHandler.html[BodyHandler]` to read `POST` requests content.

=== GraphiQL client

As you are building your application, testing your GraphQL queries in https://github.com/graphql/graphiql[GraphiQL] can be handy.

The GraphiQL user interface is disabled by default.
To enable it, configure the `link:../dataobjects.html#GraphiQLOptions[GraphiQLOptions]`:

[source,ruby]
----
require 'vertx-web-graphql/graph_ql_handler'
options = {
  'graphiQLOptions' => {
    'enabled' => true
  }
}

router.route("/graphql").handler(&VertxWebGraphql::GraphQLHandler.create(graphQL, options).method(:handle))

----

Then browse to http://localhost:8080/graphql.

TIP: GraphiQL user interface is enabled automatically when Vert.x Web is in development mode.
To switch the development mode on, use the `VERTXWEB_ENVIRONMENT` environment variable or `vertxweb.environment` system property and set it to `dev`.

Please refer to the `link:../dataobjects.html#GraphiQLOptions[GraphiQLOptions]` documentation for details on customization options.

If your application is protected by authentication, you can customize the headers to send by GraphiQL dynamically:

[source,ruby]
----
require 'vertx/multi_map'
require 'vertx-web-graphql/graph_ql_handler'
options = {
  'graphiQLOptions' => {
    'enabled' => true
  }
}

graphQLHandler = VertxWebGraphql::GraphQLHandler.create(graphQL, options).graphi_ql_request_headers(lambda { |rc|
  token = rc.get("token")
  return Vertx::MultiMap.case_insensitive_multi_map().add(Java::IoVertxCoreHttp::HttpHeaders::AUTHORIZATION, "Bearer #{token}")
})

----

=== Enable query batching

Query batching consists in posting an array instead of a single object to the GraphQL endpoint.

Vert.x Web GraphQL can handle such requests but by default the feature is disabled.
To enable it, create the `link:../../yardoc/VertxWebGraphql/GraphQLHandler.html[GraphQLHandler]` with options:

[source,ruby]
----
require 'vertx-web-graphql/graph_ql_handler'
options = {
  'requestBatchingEnabled' => true
}

handler = VertxWebGraphql::GraphQLHandler.create(graphQL, options)

----

=== Providing data fetchers with some context

Very often, the `link:../../yardoc/VertxWebGraphql/GraphQLHandler.html[GraphQLHandler]` will be declared after other route handlers.
For example, you could protect your application with authentication.

In this case, it is likely that your data fetchers will need to know which user is logged-in to narrow down the results.
Let's say your authentication layer stores a `User` object in the `link:../../yardoc/VertxWeb/RoutingContext.html[RoutingContext]`.

You may retrieve this object by inspecting the `DataFetchingEnvironment`:

[source,java]
----
dataFetcher = Java::IoVertxExtWebHandlerGraphql::VertxDataFetcher.new(lambda { |environment,future|

  routingContext = environment.get_context()

  user = routingContext.get("user")

  retrieve_links_posted_by(user, &future)

})

----

NOTE: The routing context is available with any kind of data fetchers, not just `link:unavailable[VertxDataFetcher]`.

If you prefer not to expose the routing context to your data fetchers, configure the GraphQL handler to customize the context object:

[source,java]
----
require 'vertx-web-graphql/graph_ql_handler'
dataFetcher = Java::IoVertxExtWebHandlerGraphql::VertxDataFetcher.new(lambda { |environment,future|

  # User as custom context object
  user = environment.get_context()

  retrieve_links_posted_by(user, &future)

})

graphQL = setup_graph_ql_java(dataFetcher)

# Customize the query context object when setting up the handler
handler = VertxWebGraphql::GraphQLHandler.create(graphQL).query_context(lambda { |routingContext|

  return routingContext.get("user")

})

router.route("/graphql").handler(&handler.method(:handle))

----

=== Using dataloaders

Dataloaders help you to load data efficiently by batching fetch requests and caching results.

First, you must configure the `GraphQL` object to use the `DataLoaderDispatcherInstrumentation`:

[source,java]
----
dispatcherInstrumentation = Java::GraphqlExecutionInstrumentationDataloader::DataLoaderDispatcherInstrumentation.new()

graphQL = Java::Graphql::GraphQL.new_graph_ql(graphQLSchema).instrumentation(dispatcherInstrumentation).build()

----

Then create a batch loader:

[source,java]
----
linksBatchLoader = lambda { |keys,environment|

  return retrieve_links_from_backend(keys, environment)

}

----

TIP: If you work with Vert.x callback-based APIs, you may use a `link:unavailable[VertxBatchLoader]` to simplify your code.
